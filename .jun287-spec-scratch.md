# JUN-287 — Browser use: extension walking skeleton and native messaging pairing

Parent: JUN-278 (PRD docs/browser-computer-use-prd.md, ADR docs/adr/0017-browser-use-via-june-extension.md).

## Objective

The attended-track walking skeleton:

- A TypeScript MV3 extension as its own package at `extension/`, built with the
  existing toolchain (Vite/TS/Biome, pnpm workspace), with a pinned manifest
  `key` so the extension id is stable for load-unpacked development.
- A small signed native-messaging shim that ships inside the app bundle (dev
  builds may run unsigned; signing wired the way existing bundled native
  helpers are signed).
- Host-manifest registration: the app writes the Chrome NativeMessagingHosts
  manifest pinning the extension id (allowed_origins) to the shim path.
- A version-handshake ping round trip: extension -> shim (Chrome native
  messaging stdio) -> app (authenticated local socket) and back. Loading the
  extension unpacked in developer mode connects it to the running app, and the
  app shows the extension as paired.

## Acceptance criteria

- Load unpacked connects; the app shows the extension as paired.
- Version handshake rejects a mismatched protocol with an update prompt.
- Host manifest pins the extension id; another extension cannot connect.
- Handshake and framing covered by tests on both halves (Rust shim, TS client).

## Protocol contract (fixed)

- Chrome native messaging framing: 4-byte little-endian length prefix + UTF-8
  JSON; 1 MiB message cap; messages stay small (large payloads are file
  references by design, later slices).
- Messages carry `{"v": <protocol version int>, "type": <string>, ...}`;
  protocol version starts at 1. On connect the extension sends `hello` with
  its protocol version and extension version; the app replies `hello_ok` or
  `hello_incompatible` (surfaced as an update prompt). Version constant in
  exactly one place per half.
- The shim is a dumb relay: stdio framing on one side, the authenticated local
  socket on the other; no policy in the shim.

## Decisions fixed by the task brief

- pnpm workspace introduced as `packages: ['.', 'extension']`, existing policy
  keys kept; root `pnpm check` / `pnpm typecheck` / `pnpm test` still gate the
  extension package.
- Shim is a second Rust `[[bin]]` in src-tauri; framing/handshake/relay logic
  lives in `os_june_lib` so the bin and `src-tauri/tests/` drive the same code.
  A bare signed Mach-O resource is acceptable for the skeleton.
- Credential delivery: the extension host writes a connection descriptor JSON
  (loopback port + token + protocol version) to a fixed filename in the app
  data dir (0600), rewritten on every listener start; the shim reads it and
  authenticates on its first frame. Token per-run, in-memory.
- Chrome stable only for host-manifest registration in this slice.
- Pairing surface: a Tauri command + a small self-contained settings component
  mounted as a single block in AgentSettingsSection.

## Out of scope (do not flag)

- Browser access grant, Settings toggle, june_browser MCP server, broker
  policy (JUN-286 owns those).
- Any change to src-tauri/src/hermes_bridge.rs (hard file boundary; its
  loopback/token pattern was deliberately copied, not reused).
- Debugger control, tabs, snapshots, page actions; Chrome Web Store packaging;
  routines/managed-browser track.

## Amendments (decisions made during the build)

- `@types/chrome` could not be installed (the `sfw` install wrapper needs an
  interactive approval this headless session cannot grant). A minimal ambient
  `extension/src/chrome.d.ts` scoped to the APIs used stands in; swapping to
  `@types/chrome` is a flagged follow-up.
- Extension test runner: a root `vitest.workspace.ts` with two projects (root
  vite.config.ts + extension/vite.config.ts) so one `pnpm test` gates both.
- Release bundling of the shim: build.rs writes a `.tauri-helper/june-nm-shim`
  placeholder to satisfy tauri-build resource validation; the macOS
  `beforeBundleCommand` (`scripts/bundle-nm-shim.sh`) replaces it with the
  compiled binary and signs it (Developer ID when APPLE_SIGNING_IDENTITY is
  set, ad-hoc otherwise) after cargo build, before bundling.
- Dev shim resolution: dev builds compile the shim next to the app binary via
  `default-run = "os-june"` + the cargo runner's plain `cargo build`; the
  registration command probes exe-sibling then resource-dir paths.
- Extension vite config is not type-checked (matches root convention: root
  tsconfig includes only `src`).
- The `hello_incompatible` reply carries `expected` (the app's protocol
  version); the extension popup shows "Update required".
- The extension popup renders inside Chrome, not the June app; the shared
  design tokens (src/styles/tokens.css) are unreachable from the separate
  package, so its stylesheet uses local values (with a local --fw-medium
  mirroring the app weight scale). Deliberate, not a token violation.
- Known limitation, deferred: dev and release apps share one Chrome host
  manifest path, so whichever app registered last owns the shim path
  (follow-up when multi-channel dev matters).
