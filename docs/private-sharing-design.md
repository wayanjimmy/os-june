# Private sharing for sessions and meeting notes (JUN-308)

Status: MVP design, implemented behind the endpoints described here.
Owner surface: June desktop app. Recipient surface: read-only browser viewer
served by june-api. Everything here is deliberately scoped to the MVP; cut
lines are listed at the end.

## Goal

An owner privately shares one June session (agent conversation) or meeting
note with specific people. Recipients read it in a browser without installing
June. June's servers and operators cannot read the shared content.

## Access model

Invite-only and account-bound, per the issue:

- The owner invites one or more email addresses.
- Each invite gets its own share link. The link's path identifies the share;
  the link's **fragment** carries that recipient's key material (see Crypto).
  The fragment never leaves the browser, so the server never sees it.
- Opening the link requires signing in with OS Accounts as the invited
  recipient. A recipient without an account signs up with the invited email
  first (existing OS Accounts email/Apple/Google/phone signup).
- The URL is a locator plus decryption material; it is **not** the
  authorization credential. Authorization is the server-side ACL: ciphertext
  and key envelopes are served only to an authenticated user whose verified
  email matches a non-revoked invite. A forwarded link therefore fails for
  an uninvited account even though it carries key material - there is
  nothing to decrypt without the ciphertext.
- The owner sees invite state (pending / accepted / revoked) and can revoke
  per recipient. Revocation immediately stops serving the ciphertext and
  envelope to that recipient. As the issue requires, the UI states that
  revocation cannot erase content a recipient already viewed or copied.

## Crypto

All cryptography runs client-side in one TypeScript module
(`src/lib/share-crypto.ts`) used by both the June app webview and the
browser viewer. WebCrypto only; no new native dependencies.

- **Content key (CK):** 256-bit random per share. The share payload
  (canonical JSON, see Payload) is encrypted with AES-256-GCM under CK with
  a random 96-bit IV.
- **Invite key (IK):** 256-bit random per invite, generated by the owner's
  client. The per-recipient **envelope** is `AES-256-GCM(CK, key = IK)`.
- **Link format:** `https://june-api.opensoftware.co/s/{share_id}#{invite_id}.{base64url(IK)}`
  The server stores `{invite_id -> envelope}`; it never sees IK or CK.
- **Server stores only:** ciphertext, IVs, envelopes, ACL metadata. There is
  no June-side key that can decrypt any share. Losing the link means the
  recipient must be re-invited (owner mints a fresh IK + envelope).

Why fragment-carried invite keys instead of wrapping to account-bound
public keys: no per-user key directory exists anywhere in the platform
today, and wrap-on-accept schemes require the owner's client to come back
online before a recipient can read. Fragment keys give real E2EE now - the
server never holds decryption material - while authorization stays fully
account-bound. If/when OS Accounts grows user keypairs, envelopes can be
re-wrapped to those keys without changing the wire model (the envelope
abstraction is already per-recipient).

Consequence the owner UX embraces: **the owner's client delivers each link**
(copy per-recipient link). June's server never sends invite email in the
MVP, because a server-sent email containing the fragment would hand the
server the key material and silently break the E2EE claim. "Resend invite"
is "copy the link again"; a server-side notification email (locator only,
no fragment) can be added later without weakening the model.

## Metadata June can see (documented per the issue)

Owner user id, invited email addresses (lowercased), recipient user id once
bound, item type (note / session), timestamps, ciphertext size, access
events (who fetched which share when), revocations. Nothing else: no titles,
no content, no keys. Titles are inside the encrypted payload.

## Payload (inside the ciphertext)

Canonical JSON, type-tagged, version-prefixed:

```json
{ "v": 1, "kind": "note",    "title": "...", "markdown": "...", "shared_at": "..." }
{ "v": 1, "kind": "session", "title": "...", "messages": [{"role": "user|assistant", "content": "..."}], "shared_at": "..." }
```

Notes share the rendered markdown (edited content when present, generated
otherwise). Sessions share the ordered `agent_messages` transcript. Audio,
attachments, tool events, and hidden system context are excluded from the
MVP payload by construction.

## Wire contract (june-api, all under OS Accounts Bearer auth)

- Wire casing is camelCase (house convention); timestamps are RFC 3339.
- `POST /v1/shares` - owner creates a share:
  `{ kind, ciphertextB64, ivB64, invites: [{ email, envelopeB64, envelopeIvB64 }] }`
  -> `{ shareId, invites: [{ inviteId, email }] }`. 10 MB ciphertext cap.
- `GET /v1/shares` - owner lists their shares (array of `{ shareId, kind, createdAt }`).
- `GET /v1/shares/{share_id}` - owner detail: invites with
  `state` (`pending` | `accepted` | `revoked`) and `lastAccessAt`.
- `POST /v1/shares/{share_id}/invites` - owner adds invites (same shape as
  create) -> new invite ids. An email that already holds a non-revoked invite
  on the share is rejected (400): the viewer authorizes by any active invite
  for a verified email, so a second active row would survive revoking the
  first. Re-inviting is allowed once the prior invite is revoked.
- `DELETE /v1/shares/{share_id}/invites/{invite_id}` - owner revokes.
- `DELETE /v1/shares/{share_id}` - owner unshares entirely.
- `GET /v1/shares/{share_id}/view?invite={invite_id}` - **recipient** fetch.
  Resolves the caller's verified emails via OS Accounts `/me` (the access JWT
  carries only `sub`), matches non-revoked invites, binds `recipient_user_id`
  on first access, records an access event, and returns
  `{ kind, ciphertextB64, ivB64, envelopeB64, envelopeIvB64 }` (the owner's
  display name travels inside the encrypted payload). The `invite` query
  parameter is the invite id from the link fragment (the key material stays
  in the fragment and never reaches the server); it pins the response to that
  invite's envelope, so a caller whose account holds more than one active
  invite (e.g. two verified emails each invited separately) gets the envelope
  for the link they opened rather than whichever active invite matches first.
  It only narrows the match (the binding/email authorization still applies).
  It is optional: an owner fetching their own share is served without an
  envelope regardless. View fetches lock the share row for the transaction,
  so an owner deletion and a recipient fetch have a single commit order: once
  deletion commits, no concurrent fetch can return ciphertext copied before
  that deletion.
- `GET /s/{share_id}` - the viewer HTML shell (no auth, no content, no
  share-existence signal; the same page is served for any well-formed id).
  `X-Robots-Tag: noindex, nofollow`, `Referrer-Policy: no-referrer`, strict
  CSP. `GET /robots.txt` disallows `/s/`.

Non-enumeration: every failure of `/view` and the owner endpoints for a
non-owned/unknown/revoked share returns the same 404 envelope
(`share_not_found`), never 403, and reveals no metadata. Share ids are 26+
chars of base64url randomness (`shr_` + 20 random bytes); invite ids
likewise (`shi_` prefix). Invite, view, and create endpoints are
rate-limited per user and per IP (in-memory token bucket; single-instance
CVM makes this sufficient for MVP). The multi-MiB create and add-invite JSON
bodies also pass header authentication and byte-weighted admission before
Axum buffers or parses them.

## Storage

june-api has been stateless until now. This feature adds the first
persistent store: a dedicated Postgres (Railway service `postgres-june`,
same pattern as the other services in the `open-software` Railway project),
accessed through a new `june-persistence` crate (sqlx, migrations in-repo,
`JUNE__SHARE__DATABASE_URL` via sealed env). Ciphertext lives in BYTEA with
a 10 MB cap - notes and session transcripts are text; audio is out of
scope. If the API boots without a configured share database, share
endpoints return 501 `sharing_unavailable` and everything else works
unchanged, so the rollout cannot regress existing deployments.

Tables: `shares` (id, owner_user_id, kind, ciphertext, iv, created_at,
deleted_at) and `share_invites` (id, share_id, email, envelope, envelope_iv,
recipient_user_id, accepted_at, revoked_at, last_access_at, created_at). A
partial unique index on `share_invites (share_id, email) WHERE revoked_at IS
NULL` enforces at most one active invite per email per share in the database,
so the uniqueness guarantee holds under concurrent add-invite requests, not
just the application-level check.

## Recipient viewer

Served by june-api (precedent: `/verify` already serves HTML). Static shell
plus one JS bundle, no external requests (CSP `default-src 'none'` with
explicit self allowances). Flow:

1. Parse `share_id` from the path and `invite_id.IK` from the fragment.
2. OS Accounts PKCE sign-in (public OAuth client "June share viewer",
   redirect back to `/s/callback`; the fragment survives via sessionStorage).
3. `GET /v1/shares/{id}/view?invite={invite_id}` with the Bearer token (the
   invite id, not the key, selects the envelope). The viewer OAuth client
   requests only `profile:read`; June API requires `credits:spend` on every
   other authenticated surface, so this browser-readable token cannot invoke
   paid June operations.
4. Decrypt envelope with IK -> CK; decrypt ciphertext with CK; render
   read-only (markdown for notes, chat transcript for sessions).
5. First-entry Granola-style prompt: "[Owner] shared this [meeting note /
   session] with you" with primary **Download June** and secondary
   **Maybe later**; dismissing shows the content with a persistent,
   non-blocking Download June header action.

Plaintext, CK, and IK never leave the page; no analytics on the viewer.

## Owner UI (June app)

Share action on the note editor and on agent sessions. The dialog:
invite-by-email field, per-invite **Copy link** (the only way key material
exists), state chips (pending / accepted / revoked), revoke, unshare. All
crypto happens in the webview via the shared TS module; Tauri commands only
move ciphertext and metadata.

## Acceptance mapping

Every acceptance criterion in JUN-308 maps to a test:

- invite + revoke round-trip: api integration tests + client tests.
- browser viewing without June: viewer E2E walkthrough.
- dismissible download prompt: viewer E2E walkthrough.
- forwarded link fails for uninvited account: api integration test
  (authenticated non-invitee gets `share_not_found` 404).
- revoked/expired/unauthenticated/uninvited cannot fetch envelope or infer
  metadata: api integration tests assert identical 404 bodies.
- no plaintext server-side: `POST /v1/shares` handler tests assert the
  stored bytes equal the submitted ciphertext and that request logs/spans
  contain no payload fields; the crypto module is exercised round-trip in
  frontend tests.
- existing content remains unshared: sharing is strictly opt-in per item;
  there is no migration and no default share record.

## Cut from MVP (deliberate)

- Server-sent invite email (would leak key material; owner delivers links).
- Invite expiry UI (schema has room; default no expiry).
- Audio and attachments in payloads.
- Key rotation and re-encryption on revoke.
- In-app viewing of shares received (browser only for recipients).
- Multi-instance rate limiting (single CVM today).

## Open-question answers taken for MVP

- Invite expiry: none by default; no owner-set expiry yet.
- Recipient identity: full OS Accounts sign-in required (any method), with
  the invited email verified on the account.
- Viewer content at launch: note markdown and session role/content
  transcript only.
- Account recovery: none for share keys - by design there is no
  June-controlled decryption path; a recipient who loses the link is
  re-invited, and an owner re-shares from source content they still hold.
